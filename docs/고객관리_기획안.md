# 고객관리(Customer Hub) 기획안

> Dangol CRM — 고객관리 모듈 1차 기획
> 작성일: 2026-04-13
> 상태: 초안

---

## 1. 배경 및 목적

### 1.1 배경
- 영업관리(Sales) 모듈은 딜 파이프라인·대시보드·커스텀 필드·AI 에이전트 통합까지 1차 완성.
- 영업에서 성사(WON)된 딜은 현재 사후 관리 없이 종결되며, 고객 라이프사이클(온보딩→활성→갱신→이탈)에 대한 데이터가 축적되지 않음.
- 기존 `CustomersPage`는 정적 목업 수준이며 영업 데이터와 단절되어 있음.

### 1.2 목적
1. 영업 성사 시점부터 이탈까지를 하나의 흐름으로 관리.
2. 영업관리와 **동일한 패턴**(대시보드 + 데이터 섹션, 커스텀 필드, 저장된 뷰, AI)으로 구성해 사용자 학습비용 최소화.
3. Dangol의 두 핵심 차별점 — **유연한 커스터마이징**과 **AI 에이전트 활용 극대화** — 을 고객관리 영역에도 동일하게 적용.

## 2. 전체 구조

영업관리와 대칭되는 이원 구조로 설계.

| 축 | 영업관리(기존) | 고객관리(신규) |
|---|---|---|
| 대시보드 | `/sales` | `/customer-dashboard` |
| 데이터 | `/dealflow/:pageId/:viewType?` | `/customers/:pageId/:viewType?` |
| 파이프라인/라이프사이클 설정 | `/settings/pipeline` | `/settings/customer-lifecycle` |
| 커스텀 필드 설정 | `/settings/fields` (Deal) | `/settings/customer-fields` |

→ 기존 엔진(`WidgetConfig`, `SavedView`, `CustomField`, `CustomKpiDef`, `ActivityLog`, 엑셀 임포트 `column-mapping`)을 **Customer 도메인으로 일반화해 재사용**.

## 3. 영업 → 고객 데이터 흐름

### 3.1 원칙
- **비파괴적 전환**: `Deal.status = WON`이 되어도 Deal 레코드는 영업관리에 그대로 남음 (매출·파이프라인 히스토리 보존).
- **양방향 링크**: Deal ↔ Customer를 ID로 연결. 고객 상세에서 원천 딜 확인 가능, 딜 상세에서 전환된 고객으로 이동 가능.
- **사용자 제어 전환(On/Off)**: 자동 전환이 아니라, 고객관리 툴 상단에 **"영업 성사 시 고객 자동 등록"** 토글 버튼을 둠.
  - ON: 이후 발생하는 WON 딜이 자동으로 고객 목록에 반영.
  - OFF: 전환 중단. 이미 등록된 고객은 그대로 유지.
  - 토글 상태는 workspace 단위 설정(`WorkspaceSettings.autoConvertWonToCustomer`)으로 저장.
- **재구매 고객 추적**: 동일 회사가 재등장하면 새 고객을 만들지 않고 **기존 Customer에 구매 이력을 누적**. 고객 상세 타임라인에서 "언제/무엇을/얼마에" 구매했는지 시점별로 확인 가능.

### 3.2 전환 규칙
1. 서버에서 `Deal.status` 변경 감지 훅(`deals.ts` 라우트 또는 별도 service). 토글이 OFF면 훅이 건너뜀.
2. 회사명(`Deal.company`) 기준으로 기존 Customer 조회:
   - **동일 회사 Customer 존재** → **재구매 고객(Returning Customer)**으로 처리. 새 Customer를 만들지 않고 기존 레코드에 `Contract` 추가 + 활동 이력 연결. `Customer.purchaseCount`·`lastPurchaseAt` 업데이트.
   - **없음** → Customer 신규 생성.
3. 복사/연결 항목:
   - 기본 필드: 회사명, 연락처(contact), 직함(position), 전화, 이메일 (기존 고객이면 비어있는 필드만 보충, 기존 값은 덮어쓰지 않음)
   - 거래 정보: `service`, `quantity`, `amount`, `date` → `Contract` 레코드로 누적 (구매 시점 이력)
   - 커스텀 필드: `customFieldValues` → 매핑 가능한 Customer 커스텀 필드로 이관
   - 활동 로그·첨부파일: Deal에 귀속된 `ActivityLog`/`AttachedFile`을 Customer에서도 조회 가능 (모델 확장)
4. 회사명이 완전히 일치하지 않는 유사 후보는 AI 에이전트가 매칭 후보를 제시, 사용자가 최종 확인.

### 3.3 스키마 변경 (최소 침습)

```prisma
model Customer {
  // 기존 필드 유지
  lifecycleStageId  String?
  lifecycleStage    CustomerLifecycleStage? @relation(...)
  healthScore       Int?     // 0-100, AI 산출
  customFieldValues String   @default("{}")
  sourceDeals       Deal[]   @relation("DealCustomer")
  contracts         Contract[]
  activityLogs      CustomerActivityLog[]
  attachedFiles     CustomerFile[]
  tags              String   @default("[]") // JSON

  // 재구매 추적
  purchaseCount     Int      @default(0)   // 누적 구매 횟수
  firstPurchaseAt   DateTime?
  lastPurchaseAt    DateTime?
  totalRevenue      Int      @default(0)   // 만원 단위 누적
}

model WorkspaceSettings {
  workspaceId                   String   @id
  autoConvertWonToCustomer      Boolean  @default(true)  // 고객관리 툴 토글
  updatedAt                     DateTime @updatedAt
}

model Deal {
  // 기존 필드 유지
  customerId String?
  customer   Customer? @relation("DealCustomer", fields: [customerId], references: [id])
}

model CustomerLifecycleStage {
  id           String @id @default(cuid())
  workspaceId  String
  name         String           // 온보딩, 활성, 휴면, 이탈 등 사용자 정의
  color        String @default("#3B82F6")
  type         String @default("ACTIVE") // ACTIVE, CHURNED
  sortOrder    Int    @default(0)
  customers    Customer[]
}

model Contract {
  id           String   @id @default(cuid())
  workspaceId  String
  customerId   String
  sourceDealId String?  // 어떤 딜에서 왔는지
  service      String   @default("")
  quantity     Int      @default(0)
  amount       Int      @default(0)  // 만원
  startDate    DateTime @default(now())
  endDate      DateTime?
  renewalStatus String  @default("ACTIVE") // ACTIVE, RENEWED, CHURNED, PENDING
  createdAt    DateTime @default(now())
}
```

변경 범위: 기존 `Customer`에 필드 5개 + 릴레이션 4개, `Deal`에 `customerId` 1개, 신규 모델 2~3개. 기존 영업 기능에는 영향 없음.

## 4. 고객관리 대시보드

### 4.1 기본 위젯 세트
1. **고객 헬스 분포** — 활발/주의/이탈위험 스택드 바 (AI 산출)
2. **리텐션/이탈률** — 월별·분기별 코호트
3. **LTV 총합 & 평균 계약 금액**
4. **갱신 예정 계약** — 30/60/90일 이내 만료 리스트
5. **라이프사이클 스테이지별 고객 수** — 퍼널 시각화
6. **업셀 기회 Top 5** — AI가 계약·활동 패턴 분석해 제안

### 4.2 엔진
- `WidgetConfig` 모델을 Customer 대시보드에도 재사용 (또는 `scope` 필드 추가로 구분).
- 모든 위젯은 사용자가 추가/제거/크기 변경 가능 — Sales 대시보드와 동일.
- `CustomKpiDef`로 고객 전용 KPI 사용자 정의 가능.

## 5. 고객관리 데이터 섹션

### 5.1 뷰 타입
- **테이블** — `DealflowPage`와 동일한 컬럼 커스터마이즈·필터·정렬·저장된 뷰
- **카드 그리드** — 현재 `CustomersPage` 스타일을 정식 뷰로 승격
- **관계도** (추후) — 회사 ↔ 담당자 ↔ 계약 관계 그래프

### 5.2 공통 기능 (영업관리 재사용)
- 저장된 뷰(`SavedView` 스코프 확장)
- 필터·정렬·그룹화
- 엑셀 임포트/익스포트 (`column-mapping` 재사용)
- 커스텀 필드(`CustomField`에 `scope: "DEAL" | "CUSTOMER"` 추가)
- 드래그 컬럼 순서/활성화

### 5.3 고객 전용 기능
- **통합 타임라인**: 원천 딜 → 계약 체결 → 활동 이력 → 갱신/이탈까지 단일 시간축.
- **계약(Contract) 관리**: 한 고객에 여러 계약 누적, 갱신 만료 리마인더.
- **세그먼트/태그**: 저장된 뷰와 별개로 "태그" 레이어 — AI 에이전트가 참조 가능한 의미 단위.
- **커뮤니케이션 허브**: 이메일·통화·미팅 로그 통합 (`ActivityLog` 확장).
- **고객 헬스 스코어**: 활동 빈도·결제·응답성 기반 AI 산출값을 컬럼·필터로 활용.

## 6. AI 에이전트 활용

| 영역 | 기능 | 설명 |
|---|---|---|
| 전환 시 | 자동 프로파일링 | 딜 메모·활동로그를 요약해 고객 프로필 자동 작성 |
| 전환 시 | 중복 고객 매칭 | 유사 회사명 감지해 머지 후보 제시 |
| 운영 중 | 헬스 스코어 산출 | 규칙 + LLM 판단 하이브리드 |
| 운영 중 | 이탈 위험 경보 | 활동 감소·응답 지연 감지 |
| 운영 중 | 업셀/크로스셀 제안 | 유사 고객 계약 패턴 기반 |
| 운영 중 | 자연어 검색·액션 | "지난달 대비 활동 줄어든 강남 고객", "갱신 임박 고객에게 체크인 메일 초안" |
| 입력 | 엑셀 스마트 매핑 | 기존 `column-mapping.ts` 재사용 |
| 입력 | 활동 자동 로깅 | 이메일·미팅 요약 자동 기록 |

## 8. 결정사항 (확정)

1. **라우트 네이밍** → `/customers/:pageId/:viewType?` (기존 경로 덮어쓰기). 대시보드는 `/customers`가 아닌 별도 경로 필요 (예: `/customer-dashboard` 또는 `/customers` 인덱스를 대시보드로, 목록은 `/customers/:pageId`).
2. **전환 방식** → 자동 전환 ON/OFF 토글 버튼. workspace 단위 설정(`WorkspaceSettings.autoConvertWonToCustomer`)으로 저장. 기본값 ON.
3. **동일 회사 중복** → 재구매 고객으로 처리. 기존 Customer에 `Contract` 누적, `purchaseCount`·`lastPurchaseAt`·`totalRevenue` 갱신. 고객 상세 타임라인에 구매 시점별 기록.
4. **Contract 엔티티** → 도입.
5. **Dealflow 코드 재사용** → 1차는 복제, 안정화 후 공통 모듈로 추출.

### 추가 결정사항 (확정)
6. **대시보드 경로** → `/customer-dashboard` (영업관리 `/sales` + `/dealflow`와 동일한 패턴).
7. **커스텀 필드** → 별도 테이블 `CustomerCustomField`로 분리. Deal용 `CustomField`와 스키마는 동일하되 고객 도메인 독립적으로 관리.

```prisma
model CustomerCustomField {
  id          String   @id @default(cuid())
  workspaceId String
  key         String
  label       String
  type        String   // text, number, select, multi-select, date, person, phone, email, file
  required    Boolean  @default(false)
  locked      Boolean  @default(false)
  options     String   @default("[]")
  visible     Boolean  @default(true)
  sortOrder   Int      @default(0)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([workspaceId])
  @@unique([workspaceId, key])
}
```

## 9. 범위 밖 (이번 단계 제외)

- 고객 포털(외부 로그인)
- 결제/인보이스 생성
- 외부 이메일/캘린더 실시간 연동 (연락처 수집은 수동·엑셀 단계까지만)
- 마케팅 자동화(시퀀스)
